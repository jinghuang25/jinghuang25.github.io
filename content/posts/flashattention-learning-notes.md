## 第一部分：背景知识

### 1. Transformer 是什么

- 2017 年《Attention is All You Need》提出的架构，现在几乎所有大模型（GPT、BERT、LLaMA）都基于它。
- 处理**序列**：文本被切成一串 **token（词元）**。
- 结构：输入序列 → 若干层堆叠的 Transformer block → 输出。每个 block 的核心是**自注意力（self-attention）** + 前馈网络（MLP）。

基础术语：

| 术语 | 英文 | 含义 |
|---|---|---|
| 词元 | Token | 文本切分的最小单位（词或子词） |
| 序列长度 | Sequence Length (N) | 一次输入多少 token，也叫上下文长度 Context Length |
| 嵌入 | Embedding | 每个 token 表示成的向量 |
| 头维度 | Head Dimension (d) | 注意力每个头用的向量维度，常见 64 / 128 |

### 2. 一句话如何变成一组向量

关键直觉：**一句话 = 一组向量 = 一个矩阵**。

```
用户输入 "今天天气怎么样"
  → 分词(tokenizer): ["今天","天气","怎么","样"]   (N=4 个 token)
  → 每个 token 查 embedding 表 → 一个 d 维向量
  → 堆叠成一个 N×d 的矩阵
```

这就是论文里 Q、K、V 形状为 **ℝ^(N×d)** 的由来——每个 token 一个向量，N 个 token 堆成矩阵。

> 注意区分两个 "query"：注意力里的 Query 是每个 token 各自的查询向量；用户"一句查询"是输入层面。方向一致但层面不同。

### 3. 注意力 Attention 在算什么

直觉：处理一句话时，每个词要"看"其他词来理解自己（如"它"指代谁）。**注意力 = 每个 token 按相关性加权汇总其他 token 的信息。**

每个 token 产生三组向量：
- **Q (Query，查询)**：我想找什么
- **K (Key，键)**：我能提供什么（被匹配）
- **V (Value，值)**：我实际携带的信息

三步计算（论文的 S、P、O）：

```
S = Q·Kᵀ        # N×N 相关性分数矩阵（每个 token 对所有 token）
P = softmax(S)  # 每行归一化成概率（关注度分布）
O = P·V         # 用权重加权求和所有 V → 每个 token 的新表示
```

关键点：**中间的 S、P 是 N×N 矩阵**（N 个 token 两两算相关性）。

#### 手算例子：问 AI "猫吃鱼"

为便于手算，把维度缩到 d=2（真实模型 d=64/128），句子 "猫 吃 鱼"（N=3）。
经投影得到每个 token 的 Q/K/V：

| token | Q（想找什么） | K（能提供什么） | V（携带的信息） |
|---|---|---|---|
| 猫 | [1, 0] | [1, 0] | [1, 0] |
| 吃 | [0, 1] | [0, 1] | [0, 1] |
| 鱼 | [1, 1] | [1, 1] | [1, 2] |

以"鱼"更新自己为例（其他 token 换一行同理）：

**① S = Q·Kᵀ（算相关性）** q₃=[1,1] 和每个 K 点积：
```
q₃·k_猫 = [1,1]·[1,0] = 1
q₃·k_吃 = [1,1]·[0,1] = 1
q₃·k_鱼 = [1,1]·[1,1] = 2
→ S₃ = [1, 1, 2]   分数越大越相关
```

**①.5 缩放** 除以 √d = √2 ≈ 1.414（防分数过大）：
```
S₃/√2 = [0.707, 0.707, 1.414]
```

**② P = softmax(S)（分数→权重）** 取指数再归一化：
```
e^0.707=2.028, e^0.707=2.028, e^1.414=4.113, 和=8.169
P₃ = [0.248, 0.248, 0.503]
→ "鱼"把 50.3% 注意力给自己，各 24.8% 给"猫""吃"
```
注意：必须知道整行的和（8.169）才能归一化 → 这就是 softmax 耦合整行、FlashAttention 用在线 softmax 的难点。

**③ O = P·V（按权重汇总）**
```
O₃ = 0.248·[1,0] + 0.248·[0,1] + 0.503·[1,2]
   = [0.751, 1.254]
```
O₃ 就是"鱼"融合了全句上下文后的新表示（掺进约 1/4 的"猫"和"吃"，主体仍是自己）。

整句而言 S、P 是 N×N（3×3），O 是 N×d（3×2）。序列越长 N×N 越大 → 内存/计算 O(N²)。

一句话记忆：**Q 找、K 被找、V 是内容；QKᵀ 算相关、softmax 变权重、乘 V 做汇总。**

相关概念：
- **Softmax**：把一行实数变成和为 1 的概率分布。需要看到**整行**才能归一化（先求整行和）→ 这就是"softmax 耦合列"、分块计算的难点。
- **多头注意力 (Multi-Head Attention)**：并行算多组 Q/K/V（如 12 个头），多角度关注。
- **掩码 (Masking)**：挡住某些位置。GPT 自回归用 **causal mask（因果掩码）**，不让 token 看到后面的，把 S 对应位置设 −∞，softmax 后为 0。
- **Dropout**：训练时随机丢弃防过拟合。

### 4. 为什么长序列是大问题

S 矩阵是 **N×N**：
- **计算量** O(N²·d)：序列翻倍，计算翻 4 倍
- **内存** O(N²)：要存 N×N 矩阵，致命

举例：N=1K 时 S 是 100 万个数；N=64K 时 S 是 **40 亿**个数，单这一个矩阵就几十 GB，显存爆掉。

这就是早期 Transformer 上下文卡在 512 / 1K 的原因。但长文档、长代码、高分辨率图像都需要长上下文。

### 5. 近似注意力 Approximate Attention

为降低 N×N 成本，一批工作把它**近似**成线性/近线性复杂度：

- **稀疏注意力 (Sparse Attention)**：每个 token 只关注一部分。代表：Longformer、BigBird、Reformer、OpenAI Sparse。
- **低秩近似 (Low-rank Approximation)**：用低秩分解近似 N×N。代表：Linformer、Performer、Linear Attention。
- **组合**：Scatterbrain、Long-Short Transformer 等。

**共同问题（FlashAttention 的切入点）**：
1. 减少的是 **FLOP（理论计算量）**，但**墙钟时间常常没变快**甚至更慢——忽略了访存开销，且不规则操作对 GPU 不友好（访问不连续、无法合并访存）。**FLOP 少 ≠ 实际快，访存才是瓶颈。**
2. 是**近似**，损失精度。

## 第二部分：FlashAttention 论文总结

### 核心问题

自注意力在序列长度 N 上是**二次复杂度**（时间和内存都 O(N²)），长序列训练又慢又耗内存。以往近似方法主要减 FLOP，但常无真实加速，因为忽略了**访存（IO）**。

作者洞察：**注意力是访存受限（memory-bound）**，瓶颈在 GPU 的 HBM 和片上 SRAM 之间反复读写 N×N 矩阵。应让算法 **IO 感知（IO-aware）**。

### 方法：FlashAttention

**精确（exact）** 注意力算法（非近似），核心是**不把 N×N 矩阵写进 HBM**。两个技术：

1. **Tiling（分块）**：Q、K、V 切块，从慢 HBM 搬到快 SRAM 逐块算。难点是 softmax 耦合整行 → 用**在线 softmax / 分块 softmax scaling**：维护每行最大值 m 和归一化和 ℓ 两个统计量，增量更新，不看整行也能算对 softmax。
2. **Recomputation（重算）**：反向传播不存 N×N 中间矩阵，只存统计量 (m, ℓ)，反向时用 SRAM 里的 Q/K/V 块重算。FLOP 变多，但省下的 HBM 访问让它反而更快（一种选择性梯度检查点）。

前向融合成**一个 CUDA kernel**（kernel fusion）：加载 → 矩阵乘 → softmax →（可选 mask/dropout）→ 矩阵乘 → 写回，避免反复读写 HBM。

### 理论分析（IO 复杂度）

| | HBM 访问次数 |
|---|---|
| 标准注意力 | Θ(Nd + N²) |
| FlashAttention | **Θ(N²d²/M)**（M=SRAM 大小，d=head 维度） |

- d²（64-128²）远小于 M（~100KB），所以 HBM 访问**少好几倍**（最多 9×）
- 证明了**下界**：所有 SRAM 尺寸下没有精确算法能渐进更少 → 该意义下最优
- 额外内存只需 **O(N)**（线性），而非 O(N²)

### 扩展：Block-Sparse FlashAttention

推广到块稀疏（近似）：只算非零块，跳过零块。IO 复杂度再乘稀疏比例 s，比稠密版快 2-4×，扩到 64K 序列。

### 实验结果

**训练加速**：
- BERT-large：比 MLPerf 1.1 记录快 **15%**
- GPT-2：比 HuggingFace 快 **3×**，比 Megatron-LM 快 1.7-1.8×
- Long-Range Arena：快 **2.4×**（块稀疏 2.8×）

**注意力本身**：比标准 PyTorch 最多快 **3×**，内存最多省 **20×**（随序列长度线性）。

**更长上下文 → 更好模型**：
- GPT-2 上下文 1K→4K，仍比 Megatron 快，困惑度好 0.7
- 长文档分类提升 6.4 分
- 首次让 Transformer 在 **Path-X（16K）** 超随机（61.4%），块稀疏版在 **Path-256（64K）** 达 63.1%

**正确性**：验证困惑度曲线和基线几乎重合，精确且数值稳定。

### 局限与未来

- 每个变体要手写 CUDA kernel，工程量大、跨架构不通用 → 希望高级语言自动编译成 IO 感知 CUDA（类比图像处理的 Halide）
- IO 感知思想可推广到其他模块、稀疏 MLP、多 GPU 注意力、核方法

### 一句话总结

> **标准注意力慢，不是因为算得多，而是因为在 HBM 上反复读写 N×N 矩阵。FlashAttention 用分块 + 在线 softmax + 重算，把整个注意力融进一个 kernel、数据留在 SRAM 里算，既精确、又快（少几倍 HBM 访问）、又省内存（O(N)）。**

## 第三部分：与 CUDA matmul 知识的连接

| CUDA matmul（前一篇笔记） | FlashAttention 对应 |
|---|---|
| GMEM/HBM 慢 | 注意力瓶颈就是反复读写 HBM 上的 N×N 矩阵 |
| SMEM/tiling 复用 | Q/K/V 分块搬进 SRAM 复用 |
| 算术强度、访存受限 | 注意力访存受限，近似法只降 FLOP 所以没用 |
| Kernel fusion | 整个注意力融成一个 kernel |

**学习主线**：CUDA matmul 教 GPU 访存优化的通用方法论 → FlashAttention 是这套方法论在 Transformer 注意力上的经典落地，也是"算子融合 (Operator Fusion)"最著名的案例。

## 第四部分：从向量到生成（模型怎么吐出答案）

### 1. 向量在层与层之间反复精炼

一层 Transformer block 内部：

```
输入向量 → [自注意力] → [前馈网络 MLP] → 输出向量
             (融合上下文)   (对每个向量做非线性变换)
```

- **注意力**：让每个 token 融合全句上下文。
- **MLP（前馈网络）**：对每个 token 向量单独做深加工（两层全连接 + 激活）。
- 每个子模块配 **残差连接 (residual) + 层归一化 (LayerNorm)**，稳定深层训练。

关键：block **堆叠很多层**（GPT-2 12/24/48 层，大模型上百层）。每过一层表示更"懂"这句话——浅层懂局部语法，深层懂语义、逻辑、意图。

```
词向量 → block1 → block2 → ... → blockL → 最终向量 (N×d)
```

### 2. 最后一层 → 预测"下一个词"

生成任务目标是**预测下一个 token**。取最后一层里**最后一个 token 的向量**，过 **LM Head（语言模型头）** 线性层，投影到整个词表：

```
最后 token 向量 [d 维] ──(× W, d×V)──► logits [V 维]
```

V = 词表大小（如 5 万）。logits 是每个词的"原始分数"。

### 3. softmax → 概率分布

对 V 个 logits 做 softmax，得到覆盖整个词表的概率分布：

```
logits [5万] ──softmax──► 概率 [5万, 和为1]
例: "。"→0.31, "吗"→0.22, "呀"→0.15, "很好"→0.08, ...
```

### 4. 采样 → 选一个词

- **贪心 (greedy)**：选概率最大的
- **采样 (sampling)**：按概率随机抽，**温度 (temperature)** 控制随机程度
- **Top-k / Top-p**：只在高概率的几个词里挑

### 5. 自回归循环 (Autoregressive)

生成的核心：刚生成的词**拼回输入**，整个序列重走网络，预测再下一个：

```
第1轮: [猫 吃 鱼]        → "吗"
第2轮: [猫 吃 鱼 吗]     → "?"
第3轮: [猫 吃 鱼 吗 ?]   → <结束符> → 停止
```

AI"一个字一个字打出来"就是这个循环：**每次预测一个 token，拼回去，再预测**，直到结束符或长度上限。

### 6. 完整链路

```
问题 "猫吃鱼"
  → 分词 + 嵌入 → N 个向量
  → 堆叠 L 层 (注意力 + MLP) 反复精炼
  → 取最后 token 向量 → LM Head → logits
  → softmax → 词表概率分布
  → 采样选一个 token → 输出
  → 拼回输入重复 (自回归) → 直到结束符
```

### 7. 与 AI Infra 的连接（性能要点）

1. **Prefill vs Decode 两阶段**：
   - **Prefill（预填充）**：首次把整句问题并行算完 → 计算密集，**算力受限**。
   - **Decode（解码）**：之后每次只生成一个 token 反复循环 → 每步计算小但反复读权重，**访存受限**。这是 FlashAttention、KV cache 的主战场。
2. **KV Cache（键值缓存）**：自回归时前面 token 的 K、V 每轮不变，缓存在显存里，每轮只算新 token 的 Q 去和缓存 K/V 做注意力。长文本生成的关键优化，但 KV cache 随序列增长带来显存压力——又是访存/内存战场。

一句话：**Transformer 不是一次算出整段答案，而是"预测下一个词 → 拼回去 → 再预测"，循环到结束。每步核心是把最后一个向量映射成词表概率分布再采样。**

## 第五部分：基座模型 vs 对话模型（为什么它会"回答"而不只是"续写"）

一个常踩的坑：前面"猫吃鱼 → 吗 → ?"的例子其实是**续写**行为，不是**回答**。要区分两种模型。

### ① 基座模型 (Base / Pretrained LM) —— 只会续写

训练目标只有一个：给定前文，预测下一个词。所以输入"猫吃鱼"，它会**接着往下写**，而不是回答你：

```
输入: 猫吃鱼
续写: ，狗啃骨头，各得其所。   ← 只是"接着写"，不是回答
```

基座模型没有"回答问题"的概念，本质是个超强的**文本接龙机**。之前例子预测出"吗""?"就是这种行为（把你的话补成一句完整的话）。

### ② 对话/指令模型 (Chat / Instruct Model) —— 会回答

平时用的 AI（ChatGPT、Claude）是在基座模型之上，再经过 **指令微调 (SFT, Supervised Fine-Tuning) + 人类反馈强化学习 (RLHF)** 训练出来的。它学会了：看到"用户提问"的格式时，应该续写出"有帮助的回答"。

关键：输入不是裸的"猫吃鱼"，而是被包进**对话模板**：

```
<|system|>你是一个有帮助的助手。
<|user|>猫吃鱼
<|assistant|>          ← 模型从这里开始续写，因为被训练过，这里续写出来的就是回答
```

### 关键洞察：机制没变，变的是"学会续写什么"

> **底层机制完全一样**——都是"预测下一个 token → 拼回去 → 再预测"的自回归循环。
> **变的不是机制，而是模型被训练成"该续写什么"。**

- 基座模型：续写 = 把文本接下去（文本接龙）
- 对话模型：续写 = 在"助手"角色后写出有用的回答

所以"回答问题"本质上仍是"续写"，只不过：
1. 输入套上了对话模板（告诉模型"现在轮到助手说话"）
2. 模型经 SFT + RLHF，学会了"助手该说的话 = 有帮助的回答"

一句话：**AI 不是"理解问题然后组织答案"，而是"在助手角色后，一个 token 一个 token 地续写出统计上最像好答案的文本"。它之所以像在回答，是 SFT + RLHF 教会的。**
